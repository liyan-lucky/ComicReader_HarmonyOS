# 线上构建说明

本文记录漫画浏览器 App 的 GitHub Actions 在线构建流程，以及本轮排错后的最终结论。

## 当前唯一构建入口

现在只使用这个入口：

```text
Actions → APP手动构建入口 → Run workflow
```

对应文件：

```text
.github/workflows/manual-build-entry.yml
```

之前的 `漫画浏览器 Linux 在线构建` workflow 因为手动点击无反应、容易误点，已经删除。后续不要再找这个入口。

## 你需要添加的两个对象

当前流程为了先保证 GitHub 能稳定派发 job，只读取 `Variables`，不要先放到 `Secrets`。

进入仓库：

```text
Settings → Secrets and variables → Actions → Variables
```

添加两个对象：

```text
HARMONYOS_SDK_URL
HARMONYOS_FULL_URL
```

### HARMONYOS_SDK_URL

用途：完整 HarmonyOS SDK 压缩包下载链接。

压缩包里建议包含：

```text
openharmony
hms
sdk-pkg.json
command-line-tools
```

如果这些目录不在压缩包根目录，脚本会尝试在子目录里自动寻找。

### HARMONYOS_FULL_URL

用途：SDK 以外的完整命令行工具、hvigor、ohpm 或补充文件压缩包下载链接。

这个对象用于解决 GitHub Linux 环境里命令行工具不完整的问题。补充包会被合并到：

```text
/home/runner/harmonyos-sdk/command-line-tools
```

## 当前流程做了什么

```text
1. 拉取源码
2. 准备 Java 17
3. 准备 Node 20
4. 安装 unzip、zip、curl、jq、python3、rsync、file
5. 调用 scripts/github_build_harmonyos_linux.sh
6. 下载完整 SDK
7. 下载 SDK 以外补充文件
8. 合并 SDK 和补充工具
9. 检查 App 项目结构
10. 尝试安装 ohpm 依赖
11. 临时改成未签名构建
12. 执行 HAP 构建
13. 上传 HAP 产物
```

## 构建脚本位置

复杂构建逻辑已经从 workflow 文件移到脚本里：

```text
scripts/github_build_harmonyos_linux.sh
```

这样做的原因是：workflow 文件越复杂，GitHub 越容易在派发阶段报错或出现“能识别但点运行没反应”的问题。现在 workflow 只负责准备环境、调用脚本和上传产物。

## 产物位置

脚本内部产物目录：

```text
/tmp/comic_reader_harmonyos_artifacts
```

Actions 上传产物名称：

```text
comic-reader-hap
```

如果构建成功，在本次运行页面底部的 Artifacts 下载。

## 为什么先做未签名构建

当前流程主要用于验证项目结构、SDK、hvigor 和 HAP 出包链路，不先卡签名。

脚本会临时把主构建配置里的：

```text
signingConfig: default
```

调整为空，先生成未签名 HAP。后续确认出包稳定后，再追加签名材料流程。

## 本轮已经踩过的坑

### 1. 手动入口能看到，但点击无反应

现象：

```text
能看到 workflow
能看到手动状态
点击 Run workflow 没反应
或者自动记录运行时间 0 秒
没有任何 job 步骤可以展开
```

判断：GitHub 识别了 workflow 文件，但没有真正创建 job。一般是 workflow 在解析或派发阶段就失败。

处理：先用最简单的 `APP手动构建入口` 验证仓库 Actions 权限正常，再把真正构建逻辑接到这个入口里。

### 2. runner.temp 不能写在 job 顶层 env

错误：

```text
Invalid workflow file
Unrecognized named-value: 'runner'
```

错误写法：

```text
jobs.build.env:
  ARTIFACT_DIR: ${{ runner.temp }}/xxx
```

原因：job 顶层 `env` 解析时还没进入 runner 环境，所以 GitHub 不认识 `runner.temp`。

当前处理：改成固定路径。

```text
ARTIFACT_DIR: /tmp/comic_reader_harmonyos_artifacts
```

上传路径也使用同一个固定目录。

### 3. 不要在入口里写太复杂的表达式

之前尝试过：

```text
secrets.HARMONYOS_SDK_URL || vars.HARMONYOS_SDK_URL
```

这类表达式容易让问题变复杂。当前为了先跑通，只读取：

```text
vars.HARMONYOS_SDK_URL
vars.HARMONYOS_FULL_URL
```

也就是只用 Variables。等出包稳定后，再考虑加入 Secrets 兼容。

### 4. workflow 文件不要写太长

长脚本都放到：

```text
scripts/github_build_harmonyos_linux.sh
```

workflow 文件只保留最少步骤。这样更接近已经验证过可识别、可手动触发的入口结构。

## 参考来源

参考了 `rustdesk_harmonyos` 仓库的 Linux 构建流程：

```text
先准备 Java 17 和 Node 20
使用完整 SDK 包
使用补充包修复 hvigor/命令行工具
设置 DEVECO_SDK_HOME、OHOS_HVIGOR_SDK_ROOT、DEVECO_TOOLS_HOME、DEVECO_NODE_EXE
补齐 LD_LIBRARY_PATH
最后执行 assembleHap
```

注意：RustDesk 流程里可以在具体 step 的 env 或上传产物路径使用 `runner.temp`，但不要把 `runner.temp` 放到 job 顶层 env。

## 失败时优先看哪里

优先看这些步骤：

```text
拉取源码
准备 Java 17
准备 Node 20
安装系统工具
执行中文构建脚本
上传 HAP 产物
```

如果连这些步骤都没有展开，说明仍然是 workflow 解析或派发阶段问题。

如果进入了“执行中文构建脚本”，再看脚本输出：

```text
没有填写 HARMONYOS_SDK_URL：说明 Variables 里第一个对象没填。
没有填写 HARMONYOS_FULL_URL：说明 Variables 里第二个对象没填。
没有找到 SDK 根目录：说明 SDK 压缩包层级或内容不对。
没有找到 ohpm：说明补充工具包里缺少 ohpm，或路径不对。
HAP 构建命令没有成功：说明 hvigor 工具不完整，或项目配置仍有结构问题。
没有找到 HAP 包：说明构建返回成功但没有实际出包，需要看 hvigor 输出。
```

## 注意

不要把 SDK 文件直接提交到仓库。SDK 用下载链接提供，构建时临时下载，产物用 Actions 附件保存。
