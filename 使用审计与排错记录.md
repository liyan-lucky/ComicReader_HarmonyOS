# App 使用审计与排错记录

本文记录漫画浏览器 App 和远程规则仓库联合使用时必须检查的点。

## 当前结论

App 侧基本链路已经具备：

```text
有默认远程规则地址
有网络权限
能读取远程 JSON
能校验规则字段
能把远程规则优先合并到本地规则列表
```

App 默认远程规则地址：

```text
https://raw.githubusercontent.com/liyan-lucky/ComicReader_Rules/main/generated/index.json
```

远程规则仓库现在已经加入手工稳定规则兜底，不再只依赖自动搜索结果。

## App 读取远程规则的流程

App 内操作：

```text
设置 → GitHub 远程规则 → 从 GitHub 更新规则
```

实际流程：

```text
读取 generated/index.json
→ 解析 rules 数组
→ 过滤缺少关键字段的无效规则
→ 把远程规则放在本地规则前面
→ 同名规则以远程规则优先
```

所以判断是否可用，不是看发布页有没有附件，而是看：

```text
generated/index.json 里面 rules 有多少条
```

## 已审计通过的点

### 1. 网络权限

App 已申请网络访问权限。没有这个权限就无法读取远程规则、公开网页、图片地址。

### 2. 默认地址

App 默认地址指向规则仓库主分支的 `generated/index.json`，这是正确地址。

### 3. 规则字段

远程规则必须至少包含：

```text
id
name
homepage
readerImageRegex
readerImageGroups
```

否则 App 会过滤掉，不会加入规则列表。

### 4. 空规则问题

如果远程索引里的 `rules` 是空数组，App 会继续使用内置本地规则，但远程规则仓库本身没有实际作用。

现在规则仓库已加入手工稳定规则兜底，远程索引当前应不再是 0 条。

## 使用时怎么判断是否正常

更新远程规则后，看设置页提示：

```text
远程规则更新成功：X 条
```

其中 X 必须大于 0。

如果 X 是 0，说明远程索引仍然没有有效规则，需要回规则仓库检查：

```text
generated/index.json
rules/manual/index.json
generated/rulebot_report.json
```

## 搜索和阅读的关系

App 里有两类规则：

```text
可搜索规则：有 searchUrl，可以通过漫画名搜索。
直达规则：searchUrl 为空，适合粘贴公开详情页或章节页地址。
```

所以远程规则不是只要存在就一定能通过漫画名搜到结果。要能参与漫画名搜索，规则必须有 `searchUrl`。

当前手工稳定规则里已放入可搜索规则和通用直达规则，避免远程更新后只有空壳。

## 线上构建审计结论

当前线上构建入口已经归一到：

```text
Actions → APP手动构建入口 → Run workflow
```

对应文件：

```text
.github/workflows/manual-build-entry.yml
```

复杂构建逻辑放在：

```text
scripts/github_build_harmonyos_linux.sh
```

原来的 `漫画浏览器 Linux 在线构建` workflow 因为点击无反应、容易误点，已经删除。后续只用 `APP手动构建入口`。

### 当前线上构建对象

当前为了先保证 GitHub 能稳定派发 job，只读取 Variables：

```text
HARMONYOS_SDK_URL
HARMONYOS_FULL_URL
```

填写位置：

```text
Settings → Secrets and variables → Actions → Variables
```

不要先放 Secrets。等出包稳定后，再考虑追加 Secrets 兼容。

### 本轮构建排错经验

#### 1. 手动入口能看到，但点击无反应

现象：

```text
能看到 workflow
能看到手动状态
点击 Run workflow 没反应
自动记录运行时间 0 秒
没有任何 job 步骤可展开
```

判断：GitHub 识别了 workflow 文件，但没有真正创建 job。一般是 workflow 解析或派发阶段就失败。

处理经验：先用极简入口验证 Actions 权限，再把真正构建逻辑接到这个入口里；不要在 workflow 文件里堆太多脚本。

#### 2. runner.temp 不能放在 job 顶层 env

错误信息：

```text
Invalid workflow file
Unrecognized named-value: 'runner'
```

原因：job 顶层 `env` 解析时还没有进入 runner 环境，所以不能写：

```text
ARTIFACT_DIR: ${{ runner.temp }}/xxx
```

当前处理：固定为：

```text
ARTIFACT_DIR: /tmp/comic_reader_harmonyos_artifacts
```

上传产物也使用：

```text
/tmp/comic_reader_harmonyos_artifacts/*
```

#### 3. 长脚本不要直接写进 workflow

复杂逻辑放到脚本文件：

```text
scripts/github_build_harmonyos_linux.sh
```

workflow 只保留拉源码、准备环境、调用脚本、上传产物。这样更容易被 GitHub 正常识别和派发。

#### 4. 参考 RustDesk 鸿蒙 Linux 构建流程

参考点：

```text
准备 Java 17
准备 Node 20
下载完整 SDK
合并 SDK 以外补充文件
设置 DEVECO_SDK_HOME、OHOS_HVIGOR_SDK_ROOT、DEVECO_TOOLS_HOME、DEVECO_NODE_EXE
补齐 LD_LIBRARY_PATH
最后执行 assembleHap
```

注意：RustDesk 流程里的 `runner.temp` 是在具体 step 的 env 或上传路径中使用，不要放到 job 顶层 env。

## 仍需要注意的限制

App 和规则仓库只处理公开可访问内容，不做这些事情：

```text
不登录
不付费
不绕验证码
不破解加密接口
不伪造专用客户端协议
不做反爬绕过
```

如果某个站点页面本身公开，但图片地址靠客户端脚本动态生成，App 会优先尝试静态解析，失败后走渲染卷轴兜底；如果站点禁止公开访问，就不能保证可读。

## 推荐排查顺序

遇到“没有结果”或“远程规则没用”时，按这个顺序查：

```text
1. App 设置页远程规则条数是否大于 0
2. 规则仓库 generated/index.json 的 rules 是否为空
3. 规则有没有 searchUrl
4. 当前搜索模式是否启用了公开 HTML 规则
5. 搜索过滤是否把未验证公开源过滤掉
6. 直接粘贴公开详情页地址是否能解析
7. 章节页是否能提取图片或进入渲染卷轴兜底
```

遇到“线上构建没反应或失败”时，按这个顺序查：

```text
1. Actions 里是否只使用 APP手动构建入口
2. workflow 是否能展开 job 步骤
3. 是否提示 Invalid workflow file
4. 是否把 HARMONYOS_SDK_URL 和 HARMONYOS_FULL_URL 放到了 Variables
5. 是否进入了 执行中文构建脚本 步骤
6. SDK 压缩包是否能找到 openharmony、hms、sdk-pkg.json
7. 补充包是否包含 hvigor、ohpm 或 command-line-tools
8. 是否生成 /tmp/comic_reader_harmonyos_artifacts 下的 HAP 包
```

## 给后续开发的经验

不要只看工作流是否成功，也不要只看发布页是否有附件。真正决定 App 能不能用的是：

```text
App 能否读取 generated/index.json
索引里的 rules 数量是否大于 0
规则字段是否能通过 App 校验
规则是否有 searchUrl 或能匹配直达公开页面
```

不要只看 Actions 里有没有 workflow 名称。真正决定线上构建有没有开始的是：

```text
运行记录里是否创建了 job
job 里是否能展开步骤
是否进入 执行中文构建脚本
是否最终上传 HAP 产物
```
